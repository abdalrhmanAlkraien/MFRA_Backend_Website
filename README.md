# Claude Project Template

A reusable AI agent automation framework for building full-stack applications
with Spring Boot, React, and Flutter — powered by OpenClaw and SpecKit.

Drop design images. Write a short page spec. The agent analyzes requirements,
generates tasks, writes code, runs tests, and tracks everything automatically.

---

## What This Template Does

```
You drop:    designs/1.png + designs/1.md
             ↓
Agent runs:  /analyze-page 1
             → reads image + spec
             → extracts DB schema, API endpoints, business rules
             → writes designs/1-requirements.md
             → generates tasks in systemTasks.md
             ↓
Agent runs:  /generate-spec consultation
             → produces specs/consultation/spec.md + plan.md + tasks.md
             ↓
Agent runs:  /execute-task
             → asks: backend / frontend / mobile / all?
             → builds entity + service + controller + migration
             → builds pages + components + RTK Query
             → runs JUnit + Playwright tests
             → writes documentation
             → updates all tracking files
```

## Supported Stack

| Layer | Technology |
|---|---|
| Backend | Spring Boot 3.x + Java 21 |
| Frontend | React 18 + TypeScript + Vite + RTK Query + Tailwind |
| Mobile | Flutter (optional — if listed in stack.md) |
| Database | PostgreSQL + Flyway |
| Cache | Redis (optional) |
| Storage | AWS S3 (optional) |
| Email | SMTP + Thymeleaf (optional) |
| Auth | JWT stateless |

All providers are optional and controlled by `project/stack.md`.
The agent only builds what is listed there.

## Setup — New Project in 4 Steps

### Step 1 — Copy the Template

```bash
cp -r claude-project-template/.claude ./your-project/.claude
cp -r claude-project-template/designs ./your-project/designs
```

### Step 2 — Fill 4 Files (~30 minutes)

**`project/stack.md`** — your technology choices:
```yaml
Backend:   Spring Boot 3.x
Frontend:  React 18 + TypeScript
Mobile:    ⛔ not in this project
Database:  PostgreSQL
Cache:     Redis
Email:     SMTP
Storage:   AWS S3
```

**`project/users.md`** — who can do what:
```
PUBLIC    → any visitor — no auth
ADMIN     → full access to admin panel
EDITOR    → content management only
```

**`project/pages.md`** — your page list with URLs and module names.

**`.claude/CLAUDE.md`** — project name, company, budget, architecture rules.

### Step 3 — Validate

```
/check-ready
```

Shows a 9-layer checklist. Fix any ❌ blocking issues before continuing.

### Step 4 — Add Designs and Start

```bash
# For each page:
cp designs/PAGE_TEMPLATE.md designs/1.md
# Fill in 1.md — business purpose, access, permissions, notes
# Drop 1.png into designs/

# Then run:
/analyze-page 1
```

```bash

```

### Running all pending tasks

```
/continue-tasks
```

Agent asks once which platforms to build, then runs every pending task
in dependency order, auto-fixes failures (2 attempts), and stops only
when done or budget is low.

### Checking project status

```
/review-progress
```

Shows progress bar, per-phase breakdown, test summary, config status, budget.

### Fixing a failed task

```
/fix-task 3.1
```

Agent reads the failure, diagnoses whether it is a code bug or a
requirements issue, fixes the right layer, updates all tracking files.

### Adding a feature mid-project

```
/add-feature "Add a partners logo grid to the home page"
```

Agent analyzes impact across all files, shows what will change, waits
for confirmation, then generates requirements and tasks.

### Updating a design after development started

```bash

designs/1.png  ← updated design

# Re-analyze
/analyze-page 1
# → shows a diff of what changed
# → only touches files built from page 1
# → never touches files from other pages
```

Copy `designs/PAGE_TEMPLATE.md` for each new page and fill in:

```markdown
# [Page Name]

## Business Description
What does this page do? What problem does it solve?

## User Access
- [ ] PUBLIC — any visitor, no login
- [ ] ADMIN — logged-in admin user
- [ ] EDITOR — logged-in editor user

## Permissions
View:   [what is visible]
Create: [what can be submitted]
Edit:   [what can be changed]
Delete: [what can be removed]
Other:  [publish, reorder, export, toggle, etc.]

## Related Pages
[page number or name] → [why related]

## Notes
Special behavior, business rules, edge cases, out-of-scope items.
```

The agent reads this file before opening the image. The more context
you provide here, the less clarifying questions it needs to ask.

## Tracking Files

### `tasks.md` — Per-Page View

Tracks every page through 5 stages:

```
| Page | Design | Requirements | Backend | Frontend | Mobile |
|------|--------|--------------|---------|----------|--------|
| 1    | ✅     | ✅           | ✅      | 🔄       | ⛔     |
| 2    | ✅     | ✅           | ⏳      | ⏳       | ⛔     |
| 3    | ✅     | ✅           | ⏳      | ⏳       | ⛔     |
```

Each page entry includes the exact list of files built — used to scope
re-analysis safely when a design is updated.

### `systemTasks.md` — Per-Task View

Tracks every task with:
- Status (⏳ Pending / 🔄 In Progress / ✅ Done / ❌ Failed)
- Source page (which image it came from)
- Dependencies
- Token usage and cost
- Test results summary

### `configurations.md` — Config Readiness

```
✅ Database          → set up in Task 1.1
✅ Security / JWT    → set up in Task 1.2
❌ Email             → needed by Page 7 (Consultation)
❌ File Storage      → needed by Page 3 (Blog editor)
```

## Quality Gates

Every task must pass all of these before being marked ✅:

**Backend:**
- `mvn clean verify` → BUILD SUCCESS, 0 failures
- All admin endpoints have `@PreAuthorize`
- Soft delete verified — `deletedAt` set, never hard delete
- All curl smoke tests verified against DB (`DATABASE_AUDIT.md`)

**Frontend:**
- `npm run build` → 0 TypeScript errors
- All admin routes wrapped in `AuthGuard`
- Loading, error, and empty states handled
- Playwright tests pass on Desktop + Tablet + Mobile

**Mobile (if in stack):**
- `flutter analyze` → 0 issues
- All widget tests pass

**Documentation:**
- `.claude/processed/Task X.Y.md` created
- `.claude/processed/Task X.Y - Test Results.md` created
- `tasks.md` updated — stage status + files built
- `systemTasks.md` updated — status + cost

## Tips

**Keep page specs concise.** The agent reads the full image — you only need
to add what is not visible: business rules, edge cases, out-of-scope items.

**Run `/check-ready` before starting.** Catches missing files, unfilled
templates, and budget issues before any work begins.

**Use `/analyze-page` for one page at a time.** It is safer and faster than
running the full analysis when you are adding pages incrementally.

**Let the agent ask clarifying questions.** If the spec says "preferred meeting
time" without format details, the agent will ask. Answer it — the answer gets
saved to `requirements.md` so it is never asked again.

**Never edit generated files manually.** `requirements.md`, `systemTasks.md`,
`tasks.md`, and `specs/` are all agent-owned. Edit `designs/[N].md` and
`project/*.md` instead — those are yours.

**The "Files Built" section in `tasks.md` is critical.** It is automatically
filled when tasks complete. Without it, updating a design image mid-project
cannot be done safely.

Before You Start
/check-ready
Validates the project is set up correctly before doing anything.
Checks 9 layers: core files exist, stack.md is filled, users.md is filled, all images have matching .md files, analysis has been run, tasks exist, budget is healthy.
Use when: Starting a new project. Starting a new day. Something feels wrong and you are not sure why.
/check-ready

Design Analysis
/analyze-designs
Analyzes ALL design images at once and generates the full project plan.
Reads every [N].png + [N].md pair in designs/, extracts DB schema, API endpoints, business rules, generates [N]-requirements.md for each page, groups tasks into phases, writes everything to systemTasks.md and tasks.md.
Use when: Starting a project for the first time after dropping all your design images.
/analyze-designs

/analyze-page [N]
Analyzes a single design image and merges its tasks into the existing plan.
Same BA workflow as /analyze-designs but targets one page only. Safe to run mid-project — only touches files for page N, never affects other pages.
Use when: Adding a new page after development has already started. Re-analyzing a page after the design image was updated.
/analyze-page 3
/analyze-page 7

Spec Generation
/generate-spec [module]
Converts extracted requirements into SpecKit spec files.
Reads all [N]-requirements.md files for a module and produces specs/<module>/spec.md (acceptance criteria), plan.md (architecture decisions), and tasks.md (task breakdown).
Use when: After /analyze-page or /analyze-designs — before running any tasks.
/generate-spec blog
/generate-spec case-studies
/generate-spec all

Development
/execute-task
Executes the next pending task — one task at a time.
Asks: backend / frontend / mobile / all? Then builds entity + service + controller + migration, builds pages + components, runs tests (with hard stop gates), writes documentation, updates tracking files.
Use when: You want control over each task — reviewing what the agent built before moving to the next one.
/execute-task

/continue-tasks
Runs ALL pending tasks automatically without stopping.
Asks platform once, then executes every pending task in dependency order. Auto-fixes failures (2 attempts per task), stops only when all tasks are done or budget runs low.
Use when: You trust the setup and want to build the entire project while you do something else.
/continue-tasks

Fixing Problems
/fix-task [id]
Fixes a specific task that failed, is blocked, or needs review.
Diagnoses whether the root cause is in the code or the requirements. Fixes the right layer, updates the requirements file if needed, re-runs tests, updates tracking files.
Use when: A specific task shows ❌ FAILED or ⚠️ NEEDS REVIEW in systemTasks.md and you know which task ID it is.
/fix-task 3.1
/fix-task

/implement-notes [N]
Implements issues you found during manual testing of a page.
Reads designs/[N]-notes.md that you filled after testing, implements every ⏳ Pending note (bug fixes, missing features, seed data, editor upgrades), skips ✅ Done notes, marks each done as it completes.
Use when: You manually tested a page, found problems, wrote them in [N]-notes.md, and want the agent to fix them all.
/implement-notes 3
/implement-notes 5

Review and Verification
/review-page [N]
Full 9-layer audit of one page — finds and fixes everything missing.
Checks 9 layers in sequence: requirements → spec → backend implementation → backend tests → frontend implementation → frontend tests → mobile → DB audit → full cycle behavioral verification (UI + API + Database simultaneously). Fixes every gap it finds. The full cycle runs the real application and verifies every user action at UI, API, and database level.
Use when: Development is done and you want to verify a page is truly complete — not just that it compiled, but that it actually works end to end.
/review-page 1
/review-page 2

/review-progress
Read-only snapshot of the entire project status.
Shows: tasks done vs pending vs failed, progress per phase, test results, budget used, configs ready vs missing, what to do next. Never modifies any file.
Use when: You want to see where the project stands without triggering any work.
/review-progress

Maintenance
/add-feature "..."
Adds a new feature or page mid-project.
Analyzes impact across all existing files, shows what will change, waits for confirmation, generates requirements and tasks for the new feature.
Use when: After launch you want to add something that was not in the original designs.
/add-feature "Add a partners logo grid to the home page"
/add-feature "Add export to CSV button on the users table"

/reset-tasks
Resets task statuses back to ⏳ Pending without deleting any code.
Resets tasks.md and systemTasks.md status fields only. Never touches source code, test files, requirements, or specs.
Use when: You want to re-run all tasks from scratch (after a big refactor, or when the agent did something wrong across multiple tasks).
/reset-tasks
