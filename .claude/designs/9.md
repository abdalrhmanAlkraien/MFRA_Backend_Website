# Testimonials List Page

## Business Description

The admin management view for all client testimonials displayed on the MFRA
public website. Testimonials are social proof assets — client quotes with
name, photo, role, company, and AWS service category.

Unlike blogs and case studies, testimonials go through a three-stage approval
workflow: DRAFT → PENDING → PUBLISHED. The "Pending Approval" stat card
at the top gives admins immediate visibility into testimonials awaiting review.

Three actions are available per row:
- **Edit (pencil → 10.png)** — view full testimonial details and update content
- **Preview (eye)** — view how the testimonial appears on the public website
- **Delete (trash)** — remove the testimonial with confirmation

The "+ Add New Testimonial" button navigates to 11.png (Create Testimonial).

## User Access

- [ ] PUBLIC — not accessible
- [x] ADMIN — full access — create, read, update, approve, publish, delete
- [x] EDITOR — can create, read, update, submit for approval — cannot approve,
  publish directly, or delete

> Unauthenticated → redirect to 1.png (Admin Login).

## Permissions

**ADMIN can:**
- View all testimonials (all statuses including Draft, Pending, Published)
- Add new testimonial → navigates to 11.png
- Edit any testimonial → navigates to 10.png
- Preview any testimonial (eye icon)
- Approve and publish testimonials (change status Pending → Published)
- Reject testimonials (change status Pending → Draft with a note)
- Delete any testimonial (with confirmation modal)
- Filter by status and category
- Download testimonials list (download icon on table header)

**EDITOR can:**
- View all testimonials
- Add new testimonial → navigates to 11.png
- Edit any testimonial → navigates to 10.png
- Submit testimonial for approval (change status Draft → Pending)
- Preview testimonials
- Cannot approve or publish directly (no Pending → Published)
- Cannot delete (action hidden)
- Cannot reject testimonials

Create: via 11.png (Add New Testimonial)
Edit:   via 10.png (Edit Testimonial details)
Delete: DELETE /api/admin/testimonials/:id (ADMIN only, with confirmation)

### API Endpoints

```
GET /api/admin/testimonials
  Query params:
    status:   draft | pending | published | all (default: all)
    category: string
    search:   string
    from:     date (ISO)
    to:       date (ISO)
    page:     number (default: 1)
    size:     number (default: 10)
    sort:     date_desc | date_asc | status
  Response: {
    data: array of testimonial summaries,
    total: number,
    published_count: number,
    pending_count: number,
    page: number,
    size: number
  }

Each testimonial summary:
  {
    id:           UUID
    client_name:  string
    client_photo_url: string | null
    job_title:    string
    company:      string
    category:     string
    excerpt:      string  (first 100 chars of quote)
    status:       DRAFT | PENDING | PUBLISHED
    created_at:   timestamp
  }

GET /api/admin/testimonials/summary
  → Returns stat card counts only (total, published, pending)
  → Called on page load to populate the 3 summary cards

DELETE /api/admin/testimonials/:id
  → Soft delete — sets deleted_at
  → ADMIN only

GET /api/admin/testimonials/export
  → Download testimonials as CSV
  Query params: same filter params as list endpoint
  Response: text/csv
```

### Business Rules

- Three-stage approval workflow: DRAFT → PENDING → PUBLISHED
    - EDITOR can move: DRAFT → PENDING (submit for approval)
    - ADMIN can move: PENDING → PUBLISHED (approve) or PENDING → DRAFT (reject)
    - ADMIN can also directly publish: DRAFT → PUBLISHED (bypassing pending)
    - No one can move backwards from PUBLISHED → PENDING
      (can only PUBLISH → DRAFT which is an unpublish/retract action)
- Only PUBLISHED testimonials appear on the public website
- Clicking the "Pending Approval" stat card filters table to status=pending
- EDITOR cannot delete testimonials — action is completely hidden
- EDITOR cannot approve/publish — status changes limited to Draft → Pending
- Client photo is optional — initials fallback avatar shown if no photo
- Excerpt is auto-generated from the full quote (first 100 chars + "...")
- Category tags are shared across the system (same tags as case studies and blogs)
- Date Added is the created_at date — not the published date
- Download exports only the currently filtered results (not all testimonials)
- Soft delete only — deleted_at set, never hard DELETE in DB
- A testimonial can only have one category (not multiple)
- The "Latest Submissions" label means sorted by created_at DESC by default

---

### Edge Cases

- No testimonials at all → empty state with "Add New Testimonial" button
  and subtitle "Start collecting client success stories."
- All filters applied with no matches → "No results" empty state with
  "Clear filters" link
- Pending count = 0 → card shows "0" (no special state)
- Pending count > 0 → card is slightly highlighted or pulsing to draw attention
  (subtle — primary left border more prominent when count > 0)
- Client photo URL broken/404 → initials fallback renders immediately
- Very long company name → truncate at 30 chars in Role & Company column
- Very long client name → truncate at 25 chars, full name in hover tooltip
- Very long category label → wraps within the chip (two-line chip as seen
  in design: "Data Lake Architecture" wraps to two lines)
- Category chip two-line wrap → chip height adjusts to fit, no truncation
- Network error on delete → show error toast, testimonial remains in list
- Export with no results → download empty CSV with headers only

---

### Out of Scope

- Bulk approve / bulk reject (single item only in MVP)
- Drag-and-drop reordering of published testimonials
  (display order managed in the individual testimonial detail — 10.png)
- Star rating on testimonials
- Sending approval notification email to the EDITOR who submitted
- Testimonial request emails sent to clients
- Embedding video testimonials
- Grouping testimonials by company
- Flagging or reporting testimonials as inappropriate
- Testimonial expiry dates
- A/B testing which testimonials perform better
