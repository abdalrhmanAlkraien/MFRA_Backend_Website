# Settings — System Configuration Page

## Business Description

The central configuration page for the MFRA Cloud Intelligence platform.
Controls site-wide settings that affect the public website, admin panel
behavior, and platform integrations.

Settings are grouped into logical sections:
1. **Site Information** — title, maintenance mode, support contact
2. **Social Connections** — LinkedIn, Facebook, Twitter/X profile URLs
3. **Tech Stack Tags** — global technology tags for filtering across content
4. **Security** — account password and two-factor authentication

All settings save together via a single "Save Settings" button.
A "Discard Changes" option resets unsaved edits.

The page title appears to be "System Configuration" (partially visible —
"...on" in the design, left side cut by sidebar).

### Settings are stored as key-value pairs

The `site_settings` table stores all settings as key-value pairs.
Agent reads `instructions/database.md` → Key-Value Settings Table Pattern.

**Keys to implement (from visible design):**

```
site_title                    → "MFRA Cloud Intelligence"
maintenance_mode              → "false" | "true"
support_email                 → "ops@mfra.cloud"
primary_phone                 → "+1 (555) 000-8888"
linkedin_url                  → "linkedin.com/company/mfra-cloud"
facebook_url                  → "facebook.com/mfra.official"
twitter_url                   → "x.com/mfra_cloud"
```

Tech stack tags are stored in a separate `tech_tags` table
(not as key-value — they are a proper collection with add/remove operations).

### API Endpoints

```
GET /api/admin/settings
  Response: {
    site_title:        string
    maintenance_mode:  boolean
    support_email:     string
    primary_phone:     string
    linkedin_url:      string
    facebook_url:      string
    twitter_url:       string
  }

PUT /api/admin/settings
  → Saves all settings in one request
  Request: same fields as GET response
  → Invalidates public settings cache on save

GET /api/admin/tech-tags
  Response: array of { id, name, icon_slug }

POST /api/admin/tech-tags
  Request: { name: string }
  → Creates new global tech tag

DELETE /api/admin/tech-tags/:id
  → Removes from global list (does not remove from existing content)

PATCH /api/admin/users/me/password
  Request: {
    current_password: string
    new_password:     string
  }
  → Updates logged-in user's own password

PATCH /api/admin/users/me/mfa
  → Enable/disable MFA for logged-in user
  Request: { enabled: boolean, method: TOTP | HARDWARE | SMS }
```

### Business Rules

- Only ADMIN can access Settings — EDITOR role sees no Settings in sidebar
- All site information fields save together in one PUT /api/admin/settings
- Maintenance mode toggle requires confirmation before applying
- Toggling maintenance mode immediately affects the public website
- Settings changes are cached: public site reads from Redis
  → PUT /api/admin/settings invalidates the settings cache key
- Social URLs are optional — can be blank
- Tech tags are global — changes affect all content modules immediately
- Removing a tech tag does not retroactively remove it from content
- Account Password and Two-Factor Auth sections affect logged-in user only
- Site Title is used across meta tags, email subjects, browser titles
- "Discard Changes" resets to last saved state (reads from API or local state cache)

### Edge Cases

- Maintenance mode ON and admin tries to navigate to public site:
  Public site is inaccessible — admin panel still works (different domain/port)
- Support email invalid format → inline error before save
- Social URL over 500 chars → truncate input at 500
- Adding duplicate tech tag → backend returns 409, show "Tag already exists"
- All social URLs empty → allowed (optional fields)
- Site title empty → validation error: "Site title is required"
- Password fields empty in Security section → no password change made
- New password same as current → inline error "New password must differ"
- MFA setup requires QR code display → modal or expanded card area
