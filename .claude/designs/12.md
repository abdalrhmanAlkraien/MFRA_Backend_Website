# Users Management Page

> **Note:** This page is Users Management (sidebar "Users" is active).
> It was linked as related to Testimonials — but the design clearly shows
> a user administration interface. Spec written as Users Management.

## Business Description

The central user administration panel for managing all admin and editor
accounts that have access to the MFRA Cloud Intelligence platform.

This is a security-critical page — ADMIN only. It controls who has access
to the entire admin panel. Editors appear here as managed accounts created
by administrators. No self-registration exists — all accounts are provisioned
by an ADMIN.

Three linked pages extend the functionality:
- **13.png** — view detailed profile and activity for a specific user
- **14.png** — edit an existing user's profile, role, and permissions
- **15.png** — create a brand new admin or editor account

The security notice at the bottom reinforces that all actions on this page
are logged under ROOT authority.

## User Access

- [ ] PUBLIC — not accessible
- [x] ADMIN — full access — view, create, edit, deactivate, delete users
- [ ] EDITOR — no access — this page is completely hidden from EDITOR role

> EDITOR users do not see "Users" in their sidebar navigation.
> Unauthenticated → redirect to 1.png (Admin Login).
> EDITOR attempting direct URL access → 403 redirect to /admin/dashboard.

## Permissions

**ADMIN can:**
- View all users (all roles, all statuses)
- Search users by name, email, or ID
- Filter by role (All Roles / Administrator / Editor)
- View user details → navigates to 13.png
- Edit user profile → navigates to 14.png
- Create new user account → navigates to 15.png
- Deactivate a user (Active → Inactive) without deleting
- Reactivate a user (Inactive → Active)
- Delete a user account (with confirmation — permanent action)
- Download the user list as CSV
- Advanced filter (filter icon)

**EDITOR:**
- No access — page and nav item completely hidden

### API Endpoints

```
GET /api/admin/users
  Query params:
    role:    administrator | editor | all (default: all)
    status:  active | inactive | all (default: all)
    search:  string (name, email, or UUID)
    page:    number (default: 1)
    size:    number (default: 10)
    sort:    created_at_desc | created_at_asc | name_asc
  Response: {
    data: array of user summaries,
    total: number,
    page:  number,
    size:  number
  }

Each user summary:
  {
    id:           UUID
    full_name:    string
    email:        string
    role:         ADMIN | EDITOR
    status:       ACTIVE | INACTIVE
    photo_url:    string | null
    created_at:   timestamp
    last_login:   timestamp | null
  }

PATCH /api/admin/users/:id
  → Deactivate or reactivate
  Request: { status: ACTIVE | INACTIVE }

DELETE /api/admin/users/:id
  → Soft delete (sets deleted_at) OR hard delete — decide in implementation
  → Requires ADMIN authentication
  → Cannot delete own account (backend enforces — 400 error)

GET /api/admin/users/export
  → Export filtered list as CSV
  → Same query params as list endpoint
  Response: text/csv

GET /api/admin/dashboard/summary
  → Reused for the 3 stat cards (total, admins, editors counts)
```

### Business Rules

- ADMIN ONLY page — EDITOR cannot access, sidebar item hidden for EDITOR role
- Cannot deactivate your own account (the currently logged-in user)
  → Deactivate icon hidden on the row matching the current user's ID
  → Backend also enforces: 400 "Cannot deactivate your own account"
- Cannot delete your own account (same rule)
- Deactivated user cannot log in → login attempt returns 401 with message:
  "Your account has been deactivated. Contact your administrator."
- Deactivated user's JWT is immediately blacklisted in Redis
- Deleting a user is irreversible (soft delete in DB but functionally permanent)
  → Confirmation modal: "Delete account for [Name]? This action is permanent
  and will revoke all access immediately."
- Email addresses must be unique across all admin users
- Total Users count in stat cards includes both ACTIVE and INACTIVE users
- Active Admins / Active Editors counts are active-only
- Last system-wide audit timestamp comes from a settings or audit API
- User rows are sorted by created_at DESC by default (newest first)
- Search is case-insensitive
- Search by ID supports partial UUID match (first 8 characters)
- Download exports only the currently filtered results

---

### Edge Cases

- Currently logged-in user's row:
    - Deactivate icon hidden
    - Delete icon hidden
    - No badge or indicator needed — just the hidden actions
- Only one admin exists and someone tries to delete them:
  → Backend prevents: 400 "Cannot delete the last administrator account"
- Deactivating the last active admin:
  → Backend prevents: 400 "Cannot deactivate the last active administrator"
- Search returns no results → "No users found" empty state with "Clear search" link
- All filters combined return no results → "No results match your filters" + Clear
- Very long name → truncate at 30 chars with ellipsis in the table
- Very long email → truncate at 35 chars — full email in hover tooltip
- 1,284 users — pagination is essential: `?page=129` shows the last page
- Network error on deactivate → error toast, status unchanged in UI
- Network error on delete → error toast, row remains in list
- User logs in from another device between their deactivation and JWT expiry:
  → Redis blacklist handles this — their active JWTs are invalidated immediately

### Out of Scope

- Role permission customization (only ADMIN and EDITOR — no custom roles)
- Bulk deactivate / bulk delete (single user actions only in MVP)
- User activity timeline (covered in 13.png detail page)
- Password reset by admin (user resets their own via forgot password on login)
- Inviting users via email link (admin creates account directly in 15.png)
- Two-factor authentication management per user
- IP allowlist per user
- Session management (view and revoke active sessions per user)
- API key management per user
- Audit log of actions taken BY a specific user (covered in 13.png)
- Impersonating a user (logging in as another admin)
- User groups or teams
- Transferring content ownership when deleting a user
