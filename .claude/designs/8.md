# Case Study Editor

## Business Description

The rich content editor for creating and editing case studies in the MFRA admin
panel. Accessed from the Case Studies List (6.png) via "New Case Study" or
"Edit" on an existing row.

Case studies are more structured than blog posts. Beyond the rich text content,
they require: client metadata (industry, customer name, country, city), AWS
technology tags, a results section with quantified metrics, a before/after
comparison table, and an architecture diagram image.

This additional structure makes case studies the most complex content type in
the admin panel. The editor is split into:
- **Left panel:** project title, content type, category, rich text body
- **Right panel:** publishing settings, project metadata, architecture diagram
- **Results section (below editor):** metric highlight cards + before/after table

All images (architecture diagram) upload directly to AWS S3 via pre-signed URLs.
Auto-save runs every 60 seconds with elapsed time shown ("2 min ago").

Two states — same component:
- **Create mode:** `/admin/case-studies/new` — all fields empty
- **Edit mode:** `/admin/case-studies/:id/edit` — fields pre-populated

## User Access

- [ ] PUBLIC — not accessible
- [x] ADMIN — full access to all fields and actions
- [x] EDITOR — full access to content fields — restricted on some settings
  (see Permissions below)

> Unauthenticated → redirect to 1.png (Admin Login).

## Permissions

**ADMIN can:**
- Edit all fields (title, content, content type, category)
- Set Publishing Status (Draft / Published / Archived)
- Toggle Featured Case Study on/off
- Set Display Priority (1–10)
- Edit all Project Metadata (industry, customer name, country, city, tags)
- Add and remove tags
- Build and edit Results section (metric cards + comparison table)
- Upload and replace Architecture Diagram
- Discard / delete the case study
- Publish Case Study

**EDITOR can:**
- Edit all content fields (title, content, content type, category)
- Set Publishing Status (Draft / Published — cannot Archive)
- Toggle Featured Case Study — NOT available (hidden for EDITOR)
- Display Priority — NOT available (hidden for EDITOR)
- Edit Project Metadata (all fields)
- Add and remove tags
- Build and edit Results section
- Upload Architecture Diagram
- Publish Case Study
- Cannot Discard/Delete (action hidden)

Create: POST /api/admin/case-studies
Edit:   PUT  /api/admin/case-studies/:id
Delete: DELETE /api/admin/case-studies/:id (ADMIN only, confirmation modal)

Other:
- Auto-save (PATCH /api/admin/case-studies/:id, debounced)
- Upload architecture diagram to S3 via pre-signed URL
- Publish Case Study → PATCH /api/admin/case-studies/:id/publish

### The Results Section (below editor, left panel)

This section is unique to case studies — not present in 5.md (Blog Editor).
It structures the quantified outcomes of the project.

**Section header:**
- Title: "The Results" (headline font, Manrope)
- No add button — the structure is fixed (3 metric cards + 1 comparison table)

---

#### Result Metric Cards (3 cards, horizontal row)

Each card has a left border accent (vertical colored line — this is the one
place in the design system where a 1px left border is used as a design element,
not as a divider). Border colors:
- Card 1: primary_container (#ff9900) — orange
- Card 2: tertiary_container (#38bdf8) — blue
- Card 3: primary_container (#ff9900) — orange (or second primary variant)

**Each card structure:**
```
[METRIC LABEL in label font, uppercase, muted]
[Value in display font — large, bold]
[Description line in body font, muted, 2 lines max]
```

**Card 1 — Cost Savings:**
- Metric Label: "COST SAVINGS"
- Value field: e.g. "42%" — editable input in the editor
- Description: "Annual cloud spend reduction" — editable text input
- Left border: orange

**Card 2 — Latency Improvement:**
- Metric Label: "LATENCY IMPROVEMENT"
- Value field: e.g. "120ms" — editable input
- Description: "P99 response time decrease" — editable
- Left border: blue (tertiary)

**Card 3 — Deployment Velocity:**
- Metric Label: "DEPLOYMENT VELOCITY"
- Value field: e.g. "3.5x" — editable input
- Description: "CI/CD pipeline throughput" — editable
- Left border: orange

**Editing the metric cards:**
- In editor mode: each field (label, value, description) is an editable input
- Label text is editable — admins choose their own metric names
- Value is a free-text field — "42%", "120ms", "3.5x", "$2.4M", etc.
- Description is a short text field (max 60 chars)
- In published/preview mode: rendered as static styled cards

**Adding/removing cards:**
- Fixed at 3 cards in MVP — no add/remove button
- All 3 must be filled before publish (validation)

---

#### Before/After Comparison Table

Shows legacy state vs modernized state for key metrics.

**Table structure:**
```
METRIC           |  LEGACY STATE    |  MODERNIZED STATE
─────────────────────────────────────────────────────────
[metric name]    |  [old value]     |  [new value — orange]
[metric name]    |  [old value]     |  [new value — orange]
```

**Column header row:**
- "METRIC", "LEGACY STATE", "MODERNIZED STATE"
- Label font, uppercase, muted
- Background: surface_container (slightly elevated from body)
- No border under header — color shift only

**Data rows:**
- Metric: body font, primary text
- Legacy State: body font, on_surface_variant (muted — represents old/bad)
- Modernized State: body font, primary_container orange (#ff9900) — highlighted
  (represents improvement — the key selling point)
- Row backgrounds alternate: surface_container / surface_container_low
- No row borders (No-Line Rule)

**Editing the comparison table in editor mode:**
- Each row is editable — click any cell to edit
- Metric name: text input
- Legacy State: text input
- Modernized State: text input — rendered orange when published
- "Add Row" button below the table — adds a new empty row
- Each row has a delete icon (×) on the right (visible on row hover)
- MVP: minimum 1 row, no maximum stated — practical limit is 6–8 rows

**Validation:**
- All three fields of a row must be filled
- Partially filled rows block publishing

  ```
  1. User selects file
  2. POST /api/admin/upload/presigned-url
     { filename, content_type, folder: "case-studies/diagrams" }
  3. Upload directly to S3 using presigned_url
  ```

### API Endpoints

```
GET /api/admin/case-studies/:id
  → Load existing case study for edit mode
  Response includes: all content fields + results + comparison table rows

POST /api/admin/case-studies
  → Create (triggered by first auto-save in create mode)
  Request: { title, content, content_type, category_id, status,
             industry, customer_name, country, city, tag_ids }

PATCH /api/admin/case-studies/:id
  → Auto-save and field updates (partial — only changed fields)

PUT /api/admin/case-studies/:id
  → Full update on explicit save

PATCH /api/admin/case-studies/:id/publish
  → Sets status = PUBLISHED, published_at = now()

DELETE /api/admin/case-studies/:id
  → Soft delete — ADMIN only

GET /api/admin/case-study-categories
  → Load category dropdown options

GET /api/admin/case-study-industries
  → Load industry dropdown options

GET /api/admin/tags?type=case-study
  → Load existing tags for autocomplete

POST /api/admin/tags
  Request: { name, type: "case-study" }
  → Create new tag on the fly

PATCH /api/admin/case-studies/:id/results
  → Save the results section (metric cards + comparison table)
  Request: {
    metric_cards: [
      { label, value, description, color_variant: PRIMARY | SECONDARY },
      ...
    ],
    comparison_rows: [
      { metric, legacy_value, modernized_value },
      ...
    ]
  }

POST /api/admin/upload/presigned-url
  Request: { filename, content_type, folder: "case-studies/diagrams" }
  Response: { presigned_url, public_url, key }
```

### Business Rules

- Title is required before publish — inline error if empty
- Content (rich text body) is required before publish
- Category is required before publish
- Industry is required before publish
- All 3 metric cards must be filled (label + value + description) before publish
- Comparison table: if any rows exist, all columns must be filled per row
- Architecture diagram is optional — publish allowed without it
- Only one Featured Case Study at a time — same conflict rule as 5.md
- Display Priority 1 = first position, 10 = last
- EDITOR cannot Archive, Feature, set Priority, or Delete
- Auto-save debounce: 3 seconds after last keystroke, minimum 60-second interval
- Auto-save elapsed format: "(just now)", "(1 min ago)", "(N min ago)"
  not absolute time like 5.md — this is the displayed difference between pages
- Tags are shared across case studies — creating a tag makes it available to all
- Content type defaults to "Case Study" in this editor — not changeable
  in MVP (the dropdown shows "Case Study" and is disabled or read-only here)
- Architecture diagram uploads to S3 folder: `case-studies/diagrams/`
- Customer name can be any text — no format validation
  (supports "FinTech Corp", "Major European Bank", "Confidential Client")

---

### Edge Cases

- Publish with missing metric card values → inline error on the empty fields
- Publish with partial comparison table row → inline error on incomplete row
- Featured conflict → confirmation modal before proceeding
- S3 upload fails for architecture diagram → error state on diagram slot,
  case study still saves (without diagram)
- Edit mode — case study deleted by another admin while editing → redirect to
  /admin/case-studies with toast: "This case study no longer exists"
- Very long customer name → truncate at 100 chars (field max)
- Browser closed mid-edit → auto-save preserves last saved state
- Network offline during auto-save → red dot "Save failed — tap to retry"
- Architecture diagram file too large (> 10MB) → client validation error
- Wrong file format → client validation error before upload starts
- Comparison table empty (no rows at all) → allowed for publish
  (table is optional; metric cards are required)

### Out of Scope

- Multiple architecture diagram uploads (one only in MVP)
- Video embeds in the rich text body
- Collaborative editing (multiple users simultaneously)
- Scheduled publishing (future date/time)
- Version history / revision tracking
- Preview in a new tab showing the public case study
- Custom URL slug editing (auto-generated from title)
- SEO meta title / meta description manual override
- Duplicating / cloning a case study
- Adding more than 3 metric result cards (fixed at 3 in MVP)
- Reordering metric result cards
- Importing structured data from external tools (CSV, Notion, etc.)
- Markdown raw mode toggle
- PDF attachment upload (architecture diagram as image only)
