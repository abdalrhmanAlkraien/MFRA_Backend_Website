# Case Studies List Page

## Business Description

The admin management view for all case studies published on the MFRA website.
Displays all case studies in a table with filtering, searching, and per-row
actions. This is the starting point for managing case study content.

Case studies are richer than blog posts — they represent real client projects
and include structured data: client name, industry, AWS services used,
challenge, solution, measurable results, and project duration. They are a
primary lead-generation asset on the public website.

Two key actions are available per row:
- **Insights (7.png)** — view performance analytics for a published case study
- **Edit (8.png)** — open the case study editor to write or update content

## User Access

- [ ] PUBLIC — not accessible
- [x] ADMIN — full access — create, read, update, delete, publish, feature
- [x] EDITOR — can create, read, update, publish — cannot delete or feature

> Unauthenticated → redirect to 1.png (Admin Login).

## Permissions

**ADMIN can:**
- View all case studies (all statuses including Draft and Archived)
- Create new case study → navigates to 8.png
- Edit any case study → navigates to 8.png
- View insights for any published case study → navigates to 7.png
- Publish / Unpublish any case study
- Feature / Unfeature a case study (mark as homepage featured)
- Delete any case study (with confirmation)
- Filter by status, industry, category

**EDITOR can:**
- View all case studies (all statuses)
- Create new case study → navigates to 8.png
- Edit any case study → navigates to 8.png
- View insights for published case studies → navigates to 7.png
- Publish / Unpublish case studies
- Cannot Feature / Unfeature (action hidden)
- Cannot Delete (action hidden)

Create: POST /api/admin/case-studies (via 8.png editor)
Edit:   PUT  /api/admin/case-studies/:id (via 8.png editor)
Delete: DELETE /api/admin/case-studies/:id (with confirmation modal, ADMIN only)

## Related Pages

- 7.md (Case Study Insights) → "Insights" action on each published row
- 8.md (Case Study Editor) → "New Case Study" button + "Edit" action on each row

### Filter and Search Bar

Horizontal bar below the header, above the table:

**Search input (left):**
- Placeholder: "Search case studies..."
- Searches across: title, client name, industry
- Surface_container_lowest background, no border
- Magnifier icon inside the input on the left
- Debounced search — triggers after 400ms of no typing

**Status filter (dropdown):**
- Options: All | Draft | Published | Archived
- Default: All
- Surface_container_lowest background

**Industry filter (dropdown):**
- Options: All | Financial Services | Healthcare | Retail | Manufacturing
  | Technology | Government | Energy | Logistics | Other
- Default: All
- Loaded from API (GET /api/admin/case-study-industries)

**Category filter (dropdown):**
- Options: All | Cloud Migration | AWS Infrastructure | AI & ML
  | Software Development | FinOps | Serverless
- Default: All
- Loaded from API (GET /api/admin/case-study-categories)

All filters work together — applying multiple filters narrows results.
Filter state is reflected in the URL as query params:
`/admin/case-studies?status=published&industry=financial`
so the filtered view can be bookmarked or shared.

### Case Studies Table

No horizontal dividers between rows — spacing only (No-Line Rule).
Alternating row backgrounds: surface_container / surface_container_low.

**Columns:**

| Column | Width | Content |
|---|---|---|
| Title | ~30% | Case study title + client name below in muted text |
| Industry | ~12% | Industry badge — pill chip, label font |
| Category | ~12% | Category chip — same style |
| Status | ~10% | Status badge: Draft (muted) / Published (primary) / Archived (error) |
| Featured | ~8% | Star icon — filled orange if featured, outline if not (ADMIN only column) |
| Views | ~8% | View count — shown only for Published items, dash for others |
| Last Updated | ~10% | Relative date — "2 days ago" |
| Actions | ~10% | Icon buttons: Edit, Insights, Delete |

**Title cell:**
- Title in primary text (body font, medium weight)
- Client name below in muted secondary text (label font)
- Clicking the title row → navigates to 8.png (Edit mode)

**Status badge colors:**
- Draft: surface_container_highest background + on_surface_variant text
- Published: primary_container/20 background + primary_container text
- Archived: error/20 background + error text

**Featured column (ADMIN only):**
- Star icon — filled orange (primary_container) if this case study is featured
- Empty star outline if not featured
- Clicking the star toggles featured status
- Only one case study can be featured at a time
- Toggling on shows a confirmation: "This will unfeature [previous title]. Continue?"
- Completely hidden from EDITOR role (column removed from table)

**Actions column — 3 icon buttons:**

1. **Edit** (pencil icon) — always visible
   → Navigates to 8.png (Case Study Editor, edit mode)
   → Available for both ADMIN and EDITOR

2. **Insights** (chart/bar icon) — only shown for Published items
   → Navigates to 7.png (Case Study Insights)
   → Available for both ADMIN and EDITOR
   → Hidden (not just disabled) for Draft and Archived items

3. **Delete** (trash icon) — ADMIN only
   → Shows confirmation modal:
   "Delete '[Case Study Title]'? This action cannot be undone."
   Confirm → DELETE /api/admin/case-studies/:id
   → Completely hidden from EDITOR role

**Row hover state:**
- Row background shifts to surface_container_highest
- Action icons become more visible (opacity increases from 60% to 100%)

### Bulk Actions (ADMIN only)

Checkbox on each row + select-all checkbox in the header.
When 1+ rows are selected, a bulk action bar appears above the table:

```
[N] selected     [Publish]  [Unpublish]  [Archive]  [Delete]
```

- Publish: sets all selected to Published
- Unpublish: sets all selected to Draft
- Archive: sets all selected to Archived
- Delete: confirmation modal then bulk delete

Bulk actions completely hidden for EDITOR role.

### API Endpoints

```
GET /api/admin/case-studies
  Query params:
    status:   draft | published | archived | all (default: all)
    industry: string
    category: string
    search:   string
    page:     number (default: 1)
    size:     number (default: 15)
    sort:     updated_at_desc | updated_at_asc | views_desc | title_asc
  Response: {
    data: array of case study summaries,
    total: number,
    page: number,
    size: number
  }

Each case study summary object:
  {
    id:           UUID
    title:        string
    client_name:  string
    industry:     string
    category:     string
    status:       DRAFT | PUBLISHED | ARCHIVED
    is_featured:  boolean
    views:        number (null if not published)
    updated_at:   timestamp
  }

GET /api/admin/case-study-industries
  → Load industry filter options

GET /api/admin/case-study-categories
  → Load category filter options

PATCH /api/admin/case-studies/:id/feature
  → Toggle featured status
  Request: { is_featured: boolean }
  → Backend automatically unfeatures previous featured case study

PATCH /api/admin/case-studies/:id/publish
  → Publish a case study

PATCH /api/admin/case-studies/:id/unpublish
  → Unpublish (back to Draft)

DELETE /api/admin/case-studies/:id
  → Soft delete — sets deleted_at

POST /api/admin/case-studies/bulk
  → Bulk action
  Request: { ids: UUID[], action: publish | unpublish | archive | delete }

```

---

### Business Rules

- Case studies exist independently from blog posts — separate table, separate API
- Only PUBLISHED case studies are visible on the public website
- Only one case study can be featured at a time — backend enforces this
- EDITOR cannot delete or feature case studies — those actions are hidden in UI
  and rejected by backend (403) even if called directly
- Filtering and search are applied server-side — not client-side filtering
- URL reflects filter state so filtered views can be bookmarked
- Insights icon only appears for PUBLISHED case studies — drafts have no analytics
- Views count shown only for PUBLISHED case studies — dash shown for others
- Soft delete only — deleted_at set, never hard DELETE in DB
- Archived case studies are not visible on the public website
- Featured case study appears in homepage hero/featured section

---

### Edge Cases

- No case studies at all → show empty state with "New Case Study" button
- All filters applied with no matches → show "No results" empty state
- Featured toggle conflict → confirmation modal before proceeding
- Bulk delete on 10+ items → single confirmation modal listing count not titles
- Publishing a case study that has no hero image → show warning toast but allow
- Network error on filter/search → show error state in table area with retry
- Very long title → truncate at 60 characters with ellipsis in table
- Very long client name → truncate at 40 characters with ellipsis

---

### Out of Scope

- Importing case studies from external sources
- Duplicating / cloning a case study
- Scheduled publishing (set future publish date)
- Exporting the case study list as CSV
- Drag-and-drop reordering of case studies
- Tag-based filtering (category filter covers this)
- Comparing multiple case studies performance
- Assigning case studies to specific team members
- Case study approval workflow (no draft review step)
