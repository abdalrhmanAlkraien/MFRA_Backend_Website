## Business Description

The form for submitting a new client testimonial into the MFRA admin panel.
Accessed from the Testimonials List (9.png) via the "+ Add New Testimonial" button.

Unlike the Blog and Case Study editors which are complex two-panel layouts
with rich text, this is a focused single-purpose form — testimonials have
fewer fields and a clear submission flow. It prioritizes speed of entry:
an admin or editor should be able to add a new testimonial in under 2 minutes.

On save, the testimonial is created as DRAFT. The user then either:
- Submits it for approval (EDITOR → Pending, awaits ADMIN review)
- Publishes it directly (ADMIN only → Published immediately)

No auto-save in create mode — the form is submitted explicitly.
(Unlike 10.md edit mode which auto-saves — the distinction is intentional:
creation is an intentional act, editing is iterative.)

## User Access

- [ ] PUBLIC — not accessible
- [x] ADMIN — full access — create, set all fields, publish directly
- [x] EDITOR — can create and submit for approval, cannot publish directly

> Unauthenticated → redirect to 1.png (Admin Login).

## Permissions

**ADMIN can:**
- Fill all fields
- Upload client photo
- Set display order
- Save as Draft
- Submit for Approval
- Publish Directly (skip approval queue)
- Discard (cancel without saving)

**EDITOR can:**
- Fill all content fields
- Upload client photo
- Save as Draft
- Submit for Approval
- Cannot publish directly
- Cannot set display order (field hidden)
- Discard (cancel without saving)

Create: POST /api/admin/testimonials

### Form Fields — In Display Order

**1. CLIENT PHOTO (optional, upload first for visual feedback):**
- Large circular upload zone (120px) centered at the top of the form
- Default state: dashed circle outline + camera icon + "Upload Photo" label
  (label font, muted, centered)
- On click: file picker
- On upload: shows thumbnail immediately
- Same S3 flow as 10.md:
  ```
  POST /api/admin/upload/presigned-url
    { filename, content_type, folder: "testimonials/photos" }
  → Upload to S3 directly
  → Store public_url in form state (saved with the POST on submit)
  ```
- Accepted: image/jpeg, image/png, image/webp
- Maximum: 2 MB
- Recommended: square crop, minimum 200×200px
- Photo is optional — form can be submitted without it
- Upload progress: animated ring around the circle (same as 10.md)

**2. CLIENT NAME (required):**
- Label: "CLIENT NAME" (label font, uppercase, muted)
- Input: text input, max 100 chars
- Placeholder: "Full name of the client"
- surface_container_lowest background, no border (No-Line Rule)
- Focus: primary ghost border glow
- Validation: required, min 2 chars

**3. JOB TITLE (required):**
- Label: "JOB TITLE" (label font, uppercase, muted)
- Input: text input, max 100 chars
- Placeholder: "e.g. CTO, Head of Data, VP Engineering"
- Required

**4. COMPANY (required):**
- Label: "COMPANY" (label font, uppercase, muted)
- Input: text input, max 100 chars
- Placeholder: "e.g. Veloce Financials, BioSync Labs"
- Required
- Note: may be anonymized — "Major European Bank" is a valid entry

**5. CATEGORY (required):**
- Label: "CATEGORY" (label font, uppercase, muted)
- Dropdown: single select
  Options: Cloud Migration | AWS Infrastructure | AI & ML
  | Data Lake Architecture | Security Compliance | FinOps | Serverless | Other
- Placeholder: "Select a category"
- Required
- surface_container_lowest background, chevron icon right

**6. TESTIMONIAL QUOTE (required):**
- Label: "TESTIMONIAL QUOTE" (label font, uppercase, muted)
- Textarea: tall, multi-line
- Placeholder: "Paste or type the client's exact quote here..."
- Min 50 chars, max 500 chars
- Character counter: "0 / 500" at bottom-right, updates live
    - 0–400 chars: on_surface_variant color
    - 401–480 chars: primary_container amber (approaching limit)
    - 481–500 chars: error red (near limit)
- Required

**7. DISPLAY ORDER (ADMIN only — hidden for EDITOR):**
- Label: "DISPLAY ORDER" (label font, uppercase, muted)
- Number input: integer, 1–999
- Default: auto-populated with the next available number
  (backend returns `next_display_order` from GET /api/admin/testimonials/next-order)
- Hint: "Controls position in the public testimonials section (1 = first)"
- Optional — if left at default, new testimonials appear at the end

### Validation Rules

All validation shown inline below the relevant field (not in a toast).

```
CLIENT NAME:
  required → "Client name is required"
  < 2 chars → "Name must be at least 2 characters"

JOB TITLE:
  required → "Job title is required"

COMPANY:
  required → "Company name is required"

CATEGORY:
  required → "Please select a category"

TESTIMONIAL QUOTE:
  required → "Please enter the testimonial quote"
  < 50 chars → "Quote must be at least 50 characters"
  > 500 chars → input blocked — counter turns red

PHOTO:
  wrong format → "Only JPEG, PNG, or WebP images are accepted"
  > 2 MB → "Photo must be smaller than 2 MB"
```

Validation fires:
- On field blur (when user leaves the field)
- On form submit attempt
- NOT on every keystroke (too aggressive for quote textarea)

### Form Action Buttons

Bottom of the form, right-aligned:

```
[Discard]   [Save as Draft]   [Submit for Approval]   [Publish Directly ← ADMIN only]
```

**Discard (tertiary, error color):**
- Text only, no background
- Confirmation modal: "Discard this testimonial? Nothing will be saved."
- On confirm: navigate to /admin/testimonials

**Save as Draft (secondary):**
- surface_container_highest bg + primary ghost border
- Saves the testimonial as DRAFT
- On success: navigates to 10.md (detail/edit of the new testimonial)
- Shows toast: "Testimonial saved as draft"

**Submit for Approval (secondary — primary border):**
- Creates the testimonial AND sets status = PENDING in one action
- Available to both ADMIN and EDITOR
- On success: navigates to /admin/testimonials (list)
- Shows toast: "Testimonial submitted for approval"

**Publish Directly (primary gradient — ADMIN only):**
- Creates the testimonial AND sets status = PUBLISHED in one action
- Hidden for EDITOR (not just disabled — removed from DOM)
- On success: navigates to /admin/testimonials (list)
- Shows toast: "Testimonial published successfully"

**Loading states:**
- Whichever button is clicked shows spinner + disabled state
- Other buttons also disabled during submission

### Success Flow Summary

```
EDITOR saves as Draft:
  POST /api/admin/testimonials { status: DRAFT }
  → redirect to /admin/testimonials/:newId (10.md)
  → toast: "Saved as draft"

EDITOR submits for approval:
  POST /api/admin/testimonials { status: PENDING }
  → redirect to /admin/testimonials (list, filtered to pending)
  → toast: "Submitted for approval"

ADMIN saves as Draft:
  POST /api/admin/testimonials { status: DRAFT }
  → redirect to /admin/testimonials/:newId (10.md)
  → toast: "Saved as draft"

ADMIN submits for approval:
  POST /api/admin/testimonials { status: PENDING }
  → redirect to /admin/testimonials/:newId (10.md)
  → toast: "Submitted for approval"

ADMIN publishes directly:
  POST /api/admin/testimonials { status: PUBLISHED }
  → redirect to /admin/testimonials (list)
  → toast: "Testimonial published successfully"
```

### API Endpoints

```
GET /api/admin/testimonials/next-order
  → Returns { next_display_order: number }
  → Called on page load to pre-populate Display Order field

POST /api/admin/testimonials
  Request: {
    client_name:      string (required)
    job_title:        string (required)
    company:          string (required)
    category:         string (required)
    quote:            string (required, 50–500 chars)
    client_photo_url: string | null (optional — S3 public URL)
    display_order:    number (optional — ADMIN only)
    status:           DRAFT | PENDING | PUBLISHED
  }
  Response: {
    id:     UUID
    status: string
    ...all fields
  }

POST /api/admin/upload/presigned-url
  Request: { filename, content_type, folder: "testimonials/photos" }
  Response: { presigned_url, public_url, key }
  → Photo uploaded to S3 before form submission
  → public_url stored in form state, submitted with POST
```

### Business Rules

- No auto-save in create mode — form submitted explicitly
  (deliberate distinction from edit mode in 10.md)
- Photo is uploaded to S3 before form submission
  If photo is uploaded but form is discarded → orphaned S3 file
  (acceptable in MVP — S3 lifecycle policy cleans up orphaned files after 7 days)
- EDITOR cannot publish directly — button hidden from DOM (not just disabled)
  Backend also enforces: 403 if EDITOR tries to POST with status=PUBLISHED
- Display order auto-populated from next available number
- All required fields validated before any API call is made
- On successful POST → activity log entry created automatically:
  "Created as [status] by [user name]"
- A testimonial with no photo is valid — initials avatar used on public site
- Category is a single value — not a multi-select
- Quote character limit enforced both client-side (blocks input at 500)
  and server-side (returns 422 with field error if over 500)

---

### Edge Cases

- Photo uploaded, then user discards form → orphaned S3 object
  (acceptable — S3 lifecycle rule cleans up after 7 days)
- Photo upload fails → error ring on avatar slot, form can still submit without photo
- Network error on POST → show error toast "Failed to save. Please try again."
  Form state preserved (user does not lose their input)
- User navigates away via sidebar before saving → browser confirm dialog
  "Leave page? Your testimonial has not been saved."
  (implemented via `beforeunload` event listener, removed after successful save)
- Duplicate testimonial (same client name + company) → no backend constraint,
  allowed (same person may give multiple testimonials over time)
- Display order conflict → same handling as 10.md (gaps allowed, alphabetical tiebreak)
- Very long paste into quote field → truncated at 500 chars on paste
  (clipboard paste event listener trims to limit)
- EDITOR tries POST with status=PUBLISHED → 403 from backend
  (should not happen via UI but must be enforced at API level)

### Out of Scope

- Bulk import of testimonials (CSV upload)
- Copying or duplicating an existing testimonial into a new one
- Saving a partially filled form as a template
- Rich text formatting in the quote (plain text only — no bold, italic, links)
- Multiple photos per testimonial (one headshot only)
- Video testimonial upload
- Embedding a video URL (YouTube/Vimeo) in the testimonial
- Sending an email to the client requesting a testimonial
- Auto-population from LinkedIn or other public profiles
- Previewing the testimonial as it would appear on the public site
  (preview available from 9.md after creation)
