# Pages Analysis Instructions — BA & PM Workflow

> This file is reusable across all projects.
> The AI agent reads this file when analyzing design images and generating tasks.
> It defines the full workflow: design image + page spec → requirements → tasks.

---

## How the Design + Spec System Works

Every page in the project has two files with the same name:

```
designs/
├── 1.png          ← design image (dropped by designer or you)
├── 1.md           ← page spec (written by you)
├── 2.png
├── 2.md
├── 3.png
├── 3.md
└── ...
```

**The image file** — the visual design. Can be a Figma export, screenshot,
wireframe, or mockup. Any image format (.png, .jpg, .webp) works.

**The .md file** — the page specification written by you. Contains business
context, user access, permissions, related pages, and notes. The agent reads
this before opening the image. The access level (public/admin) is defined
inside the .md file — not by folder structure.

After analysis, the agent creates a third file:
```
designs/1-requirements.md   ← generated by agent after analysis
```

---

## The .md File — What You Write Per Page

Use `designs/PAGE_TEMPLATE.md` as your starting point. Fill in:

```markdown
# [Page Name]

## Business Description
What does this page do? What problem does it solve?

## User Access
Who can see and use this page:
  PUBLIC / ADMIN / EDITOR / ADMIN+EDITOR

## Permissions
What the user on this page can do:
  View:   [what is visible]
  Create: [what can be submitted or created]
  Edit:   [what can be modified]
  Delete: [what can be removed]
  Other:  [publish, reorder, export, toggle, etc.]

## Related Pages
Pages that link to or from this page:
  [page name or number] → [why related]

## Notes
Any extra context:
  - Special behavior not visible in the design
  - Business rules the agent needs to know
  - Edge cases to handle
  - Things explicitly NOT to implement (out of scope)
```

## The Agent Workflow — Step by Step

---

### Step 1 — Read the .md File First

```
Open designs/[N].md

Extract:
  → Business purpose of this page
  → Who can access it (PUBLIC / ADMIN / EDITOR)
  → What actions they can perform
  → Which pages are related
  → Any notes or out-of-scope items

Do not open the image yet.
Build context first.
```

---

### Step 2 — Open and Read the Full Design Image

```
Open designs/[N].png

Before extracting anything:
  → Read the entire page top to bottom
  → Understand the layout: header, content, sidebar, footer, modals
  → Identify the primary action (most important user action on this page)
  → Count and name each section in the design
  → Note all visible states: normal, loading, empty, error, success
```

---

### Step 3 — Extract Every UI Element Section by Section

```
For each section in the design:

  Name it clearly:
    Hero / Filter Bar / Cards Grid / Form / Table / Modal / Sidebar / Footer

  List all elements:

    Static elements:
      → Headings, labels, body text, icons, images
      → Is this hardcoded or from API?

    Dynamic elements (data shown):
      → What field? (title, date, status, count, price)
      → Where does it come from? (which DB table)

    Interactive elements:
      → Buttons → what do they do? what API call?
      → Inputs → what do they accept? what validation?
      → Filters → what options? do they update URL params?
      → Dropdowns → static options or loaded from API?
      → Links → where do they navigate?
      → Toggles/checkboxes → what state do they change?

    States:
      → Loading → skeleton or spinner
      → Empty → what message or illustration
      → Error → what message on API failure
      → Success → what shows after form submit
```

---

### Step 4 — Map Every Visual Element to the Database

For every piece of data visible in the design:

```
Question 1 — Does this data exist in a defined table already?
  YES → reference that table (no new table needed)
  NO  → define new table or add column

Question 2 — What data type?
  Short text      → VARCHAR(N)
  Long content    → TEXT
  Integer number  → INT
  Decimal/money   → NUMERIC(19,4) — never FLOAT
  True/false      → BOOLEAN
  Date and time   → TIMESTAMPTZ — never TIMESTAMP
  UUID reference  → UUID (foreign key)
  Status or enum  → VARCHAR(50)
  Flexible JSON   → JSONB (only when structure is truly unknown)

Question 3 — Does this field need a DB index?
  Used in WHERE filter  → YES — add index
  Used in ORDER BY      → YES — add index
  Used in text search   → YES — GIN index
  Only displayed        → NO — no index needed

Question 4 — What relationships exist?
  One blog → one category     → foreign key on blogs.category_id
  One blog → many tags        → junction table blog_tag_map
  One item → many sub-items   → child table with parent_id

Question 5 — What constraints apply?
  Always required      → NOT NULL
  Must be unique       → UNIQUE constraint
  Limited valid values → VARCHAR + CHECK constraint
  Auto-generated       → DEFAULT value
```

---

### Step 5 — Write Flyway Migration SQL

For every new table, produce the Flyway SQL migration:

```sql
-- File: VX__create_<module>_tables.sql

CREATE TABLE <table_name> (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),

    -- all fields from design --

    -- status column if design shows status/badge --
    status       VARCHAR(50) NOT NULL DEFAULT '<default_value>',

    -- foreign keys for relationships --
    <parent>_id  UUID REFERENCES <parent_table>(id),

    -- audit columns — mandatory on every table --
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_by   UUID,
    updated_by   UUID,
    deleted_at   TIMESTAMPTZ    -- NULL = active, non-null = soft deleted
);

-- Index every column used in WHERE, JOIN, ORDER BY, or filter UI
CREATE INDEX idx_<table>_<column>  ON <table_name>(<column>);
CREATE INDEX idx_<table>_active    ON <table_name>(deleted_at)
    WHERE deleted_at IS NULL;
```

**Rules the agent must never break:**
```
✅ UUID primary key — never auto-increment integer
✅ deleted_at on every table — soft delete is mandatory
✅ TIMESTAMPTZ — never TIMESTAMP
✅ NUMERIC(19,4) for money — never FLOAT or DOUBLE
✅ Index every filtered or searched column
✅ NOT NULL on required fields
❌ Never use FLOAT for numbers that involve calculations
❌ Never skip deleted_at
❌ Never hard delete — set deleted_at = now() only
```

### Step 6 — Define API Endpoints

For every user action in the design, define an endpoint:

```
Design action                      → Endpoint
─────────────────────────────────────────────────────────────────
User views a list                  → GET    /api/public/<resource>
User views a single item           → GET    /api/public/<resource>/{slug}
User filters or searches           → GET    /api/public/<resource>?param=value
User loads filter options          → GET    /api/public/<resource>-categories
User submits a public form         → POST   /api/public/<resource>

Admin views all items              → GET    /api/admin/<resource>
Admin views single item            → GET    /api/admin/<resource>/{id}
Admin creates                      → POST   /api/admin/<resource>
Admin full update                  → PUT    /api/admin/<resource>/{id}
Admin partial update               → PATCH  /api/admin/<resource>/{id}
Admin deletes                      → DELETE /api/admin/<resource>/{id}
Admin publishes                    → PATCH  /api/admin/<resource>/{id}/publish
Admin unpublishes                  → PATCH  /api/admin/<resource>/{id}/unpublish
Admin toggles active               → PATCH  /api/admin/<resource>/{id}/toggle
Admin reorders items               → PATCH  /api/admin/<resource>/reorder
Admin changes status               → PATCH  /api/admin/<resource>/{id}/status
```

For each endpoint, define:
```
Auth:        None / ROLE_ADMIN / ROLE_EDITOR / ROLE_ADMIN+ROLE_EDITOR
Paginated:   yes/no — default page size
Filters:     which query params are supported
Cached:      yes/no — TTL in minutes/hours
Rate limit:  yes/no — max requests per window
```

---

### Step 7 — Extract Business Rules

```
Validation rules:
  → Required fields — which cannot be empty
  → Length rules — max and min character counts
  → Format rules — email, URL, phone number format
  → Uniqueness — which fields must be unique (slug, email)

Computed fields (auto-generated, not user-entered):
  → Slug: generated from title — lowercase, hyphens, URL-safe
  → Reading time: word count ÷ 200 (words per minute)
  → Published at: timestamp set when status → PUBLISHED
  → Created at / Updated at: set by JPA automatically

Visibility rules:
  → PUBLIC only sees PUBLISHED or ACTIVE records
  → Drafts never appear in public API responses
  → Soft-deleted records never appear in any query
  → Admin sees everything including drafts

Trigger rules:
  → On publish → invalidate public cache for this resource
  → On form submit → rate limit check → save → send email (async)
  → On delete → soft delete only (set deleted_at = now())
  → On settings update → invalidate settings cache
```

### Step 8 — Ask Clarifying Questions

If the design or .md file has ambiguity, stop and ask before generating tasks.

Format:
```
❓ CLARIFICATION NEEDED — Page [N]: [Page Name]
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[Quote the exact ambiguity from the design]

Options:
  A) [interpretation A and its implication]
  B) [interpretation B and its implication]

Waiting for your answer before generating tasks.
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

Common clarifying questions:
- Does this filter update the URL so it can be bookmarked?
- Is there a maximum number of featured items?
- Does Load More append to the list or replace it?
- Is this field a date picker or free text input?
- What is shown when this section has no data?
- Should deleted items appear in activity history?

### Step 10 — Generate Tasks

After the requirements document is complete, generate tasks.

**Always check `project/stack.md` first.** Only generate tasks for platforms listed there.

**Task format:**
```
Task [Phase].[N] — [Platform]: [Module] — [What is delivered]

  Spec:         specs/<module>/spec.md
  Depends on:   [task IDs this depends on]
  Complexity:   Simple / Medium / Complex
  Platform:     Backend / Frontend / Mobile

  Delivers:
    → [concrete file or feature]
    → [concrete file or feature]

  Endpoints:
    → [METHOD] /api/path

  Tables:
    → [table name]

  Tests:
    → Backend:  [N] scenarios
    → Frontend: [N] scenarios
    → Mobile:   [N] scenarios (only if Flutter in stack.md)
```

**Complexity guide:**
```
Simple:   1–2 entities, 2–4 endpoints, basic CRUD, 1 simple page
          Example: Stats page, Settings page, Tools page

Medium:   2–3 entities, 4–8 endpoints, filters + search, standard forms
          Example: Blog list + detail, Contact module, Consultation form

Complex:  Multiple related entities, 8+ endpoints, rich text editor,
          file uploads, async email, multi-tab forms, drag-and-drop
          Example: Blog editor, Case Study multi-tab editor
```

## Quality Checklist Before Writing Tasks

```
✅ Read the .md file before opening the image
✅ Read the full image before extracting elements
✅ Every visual data field maps to a DB column
✅ Every user action maps to an API endpoint
✅ Every filter has a corresponding DB index
✅ Every form field has a validation rule
✅ Every status badge has a status column
✅ deleted_at present on every new table
✅ TIMESTAMPTZ used — not TIMESTAMP
✅ NUMERIC(19,4) for money — not FLOAT
✅ Flyway SQL written for every new table
✅ Clarifying questions asked and answered
✅ stack.md checked — tasks only for listed platforms
✅ Backend task created before frontend task for same module
✅ Requirements document saved to designs/[N]-requirements.md
✅ tasks.md updated with all stages and sub-tasks
```

---

## ✅ Always Do This

1. Read .md file before opening the image — context first
2. Read the full image before extracting anything
3. Map every visual element to a DB column
4. Write one endpoint per user action
5. Produce Flyway SQL for every new table
6. Ask clarifying questions before generating tasks for ambiguous designs
7. Save requirements to `designs/[N]-requirements.md`
8. Update `.claude/tasks.md` after every page analysis
9. Check `stack.md` before generating any task
10. Always create backend task before the corresponding frontend task

---

## ❌ Never Do This

1. Open the image without reading the .md file first
2. Generate frontend tasks before backend tasks for the same module
3. Generate mobile tasks if Flutter is not in `stack.md`
4. Skip the DB schema — every visual field must map to a column
5. Use FLOAT for money — always NUMERIC(19,4)
6. Skip deleted_at on any table
7. Generate tasks without the requirements document first
8. Assume a design element is optional if it appears visually
9. Skip clarifying questions — always resolve ambiguity before proceeding
10. Update `systemTasks.md` without also updating `tasks.md`
